# Class Diagram Generator Modern UI Redesign Design

**작성일:** 2026-05-20

## 1. 목적

`class-diagram-generator`의 `upload`, `progress`, `result` 화면을 Bootstrap 5 기반 위에서 재구성해, 하나의 제품처럼 일관된 정보 구조와 시각 시스템을 제공한다.

이번 작업은 단순 색상 교체가 아니라 다음을 포함하는 디자인 시스템 재정비다.

- 공통 토큰 레이어 도입
- 공통 상단 앱 셸 정리
- 카드, 타임라인, 드롭존, 결과 카드 등 상태 컴포넌트 정비
- 모바일 환경까지 고려한 반응형 재배치
- 결과 화면 상단 요약 정보 확장

## 2. 범위

### 포함 범위

- `src/main/resources/templates/upload.html`
- `src/main/resources/templates/progress.html`
- `src/main/resources/templates/result.html`
- `src/main/resources/static/css/app.css`
- 신규 디자인 시스템 CSS 파일 추가
- 업로드/진행/결과 화면에서 사용하는 프론트엔드 JS의 최소 보정
- 결과 화면 요약 정보 확장을 위한 백엔드 DTO 및 저장 모델 수정
- 신규 문구에 대한 다국어 메시지 추가

### 제외 범위

- 업로드, 진행, 결과 생성의 핵심 비즈니스 로직 변경
- 산출물 생성 포맷 추가
- 인증, 사용자 계정, 작업 이력 목록 등 신규 기능 추가
- Bootstrap 제거 또는 프레임워크 교체

## 3. 현재 화면 진단

### 공통 관찰

- 세 화면 모두 연한 인디고 계열을 일부 쓰고 있으나, 공통 디자인 시스템보다는 페이지별 스타일 조합에 가깝다.
- 카드, 상태, 액션의 위계가 충분히 분리되지 않아 화면 목적이 첫눈에 또렷하지 않다.
- 공통 헤더 구조가 단순해 제품 레벨의 일관성이 약하다.
- 현재 CSS가 단일 파일에 집중되어 있어, 이후 화면 확장 시 유지보수 비용이 커질 가능성이 높다.

### 업로드 화면

- 하나의 큰 카드 안에 모든 입력이 평평하게 들어 있어 입력 순서가 직관적이지 않다.
- `프로그램 정보`, `출력 설정`, `파일 업로드`가 서로 다른 작업 성격임에도 동일한 밀도로 배치되어 있다.
- 드롭존은 기능적으로 충분하지만 화면 내 핵심 행동 요소로 보일 만큼 강조되지는 않는다.

### 진행 화면

- 퍼센트, 타임라인, 경고 영역이 존재하지만 화면 구성이 다소 비어 보인다.
- 타임라인은 데스크톱에서는 무난하지만 모바일에서 가로 흐름이 답답해질 가능성이 높다.
- 경고가 없을 때 상태 영역이 빈 느낌을 주어 화면 완성도가 떨어진다.

### 결과 화면

- 상단 완료 영역과 하단 산출물 영역의 역할 분리는 되어 있으나, 요약 정보가 최소 수준이라 결과의 전체 맥락이 약하다.
- 산출물 카드는 유지 가치가 높지만, 상단 요약과 경고 표시가 더 강화될 필요가 있다.

## 4. 사용자 승인 결정 사항

- 작업 강도는 "정보 구조 일부 재편 + 공통 디자인 시스템 도입" 수준으로 진행한다.
- 업로드 화면에서 `프로그램명/버전/언어`는 한 카드에 묶는다.
- 업로드 화면에서 `출력 형식/다이어그램 포함`은 별도 카드로 분리한다.
- 결과 화면에서는 산출물 카드형 레이아웃을 유지한다.
- 결과 화면 상단 요약에는 `생성 시각`, `만료 시각`, `총 산출물 수`를 포함한다.

## 5. 디자인 원칙

### 5.1 시각 방향

- 배경은 차분한 `slate` 계열을 사용한다.
- 기본 포인트는 `indigo`, 제한적인 보조 포인트는 `violet`를 사용한다.
- 카드와 입력 컨트롤은 밝고 정돈된 표면 위에 배치한다.
- 그라디언트는 필요한 영역에만 절제해서 사용한다.
- 컬러 그림자 대신 중성 그림자를 사용한다.

### 5.2 정보 위계

- 각 페이지는 `상단 hero + 본문 primary surface` 구조를 갖는다.
- 가장 중요한 액션은 각 화면에서 한 개만 1차 시각 강조를 받는다.
- 보조 정보는 `meta card`, `subtle surface`, `status note` 형태로 분리한다.

### 5.3 상호작용 원칙

- 입력/선택/상태/다운로드는 서로 다른 시각 패턴을 갖는다.
- 드롭존, 타임라인, 결과 카드는 루트 상태 클래스 기반으로 시각 상태를 전환한다.
- 포커스 링은 유지하고, 키보드 접근성을 해치지 않는다.

### 5.4 반응형 원칙

- 데스크톱에서는 2열 레이아웃을 적극 사용한다.
- 태블릿 이하에서는 1열 스택으로 전환한다.
- 모든 그리드는 `minmax(0, 1fr)`와 자식 `min-width: 0` 규칙을 사용해 수평 오버플로를 막는다.
- 375px 폭에서 수평 스크롤이 없어야 한다.

## 6. 공통 디자인 시스템 설계

## 6.1 CSS 구조

다음 구조로 CSS를 분리한다.

- `src/main/resources/static/css/mmu/tokens.css`
- `src/main/resources/static/css/mmu/components.css`
- `src/main/resources/static/css/app.css`

역할은 다음과 같다.

- `tokens.css`: 색상, 반경, 간격, 그림자, 타이포, 테마 토큰
- `components.css`: 공통 카드, 셸, 드롭존, 타임라인, 메타 카드, 결과 카드, 버튼 패턴
- `app.css`: 프로젝트 특화 보정과 Bootstrap 재스킨, 페이지별 소량 보완

로드 순서는 다음과 같다.

1. Bootstrap CSS
2. Bootstrap Icons
3. `tokens.css`
4. `components.css`
5. `app.css`

## 6.2 공통 셸

- 상단에는 고정형 `topbar shell`을 둔다.
- 헤더 구조는 `브랜드 / 현재 페이지 맥락 / 우측 툴바` 순서를 따른다.
- 우측 툴바에는 `언어 토글`과 `테마 토글`을 둔다.
- 모바일에서는 브랜드와 툴바가 줄바꿈 없이 유지되도록 폭과 간격을 조정한다.
- 헤더는 반투명 배경과 blur를 유지하되 현재보다 더 정돈된 패딩과 경계선을 사용한다.

## 6.3 토큰

핵심 토큰은 다음 범주를 갖는다.

- 색상:
  - 배경
  - 표면
  - 경계선
  - 본문 텍스트
  - 보조 텍스트
  - 강조색
  - 성공/경고/위험
- 형태:
  - `radius-sm`, `radius-md`, `radius-lg`, `radius-xl`
- 그림자:
  - `shadow-soft`, `shadow-md`
- 간격:
  - `space-2`, `space-3`, `space-4`, `space-6`, `space-8`
- 포커스:
  - `focus-ring-color`

## 6.4 다크 모드

- 테마는 `<html data-theme="light|dark">` 기반으로 전환한다.
- 초기 테마 결정 스크립트를 `<head>`에 인라인 삽입해 깜빡임을 줄인다.
- 사용자 선택은 `localStorage`에 저장한다.
- 다크 모드에서도 Bootstrap 기본 포커스 가시성은 유지한다.

## 6.5 Bootstrap 5 재스킨

다음 요소는 indigo 중심으로 재스킨한다.

- `.btn-primary`
- `.btn-outline-primary`
- `.form-control:focus`
- `.form-select:focus`
- `.form-check-input:focus`
- `.form-check-input:checked`

## 7. 페이지별 상세 설계

## 7.1 Upload 화면

### 목표

사용자가 업로드 전 입력해야 할 정보와 실제 핵심 행동인 ZIP 업로드를 빠르게 구분할 수 있게 만든다.

### 레이아웃

- 상단 `hero`:
  - 제목
  - 한 줄 설명
  - 업로드 대상과 생성 결과를 요약하는 짧은 보조 문구
- 본문:
  - 데스크톱 2열
  - 모바일 1열

데스크톱 기준 구성은 다음과 같다.

- 좌측:
  - `프로젝트 정보` 카드
  - `출력 설정` 카드
- 우측:
  - `소스 업로드` 카드
  - `실행 액션` 영역

### 프로젝트 정보 카드

포함 항목:

- `프로그램명`
- `버전`
- `산출물 언어`

규칙:

- 데스크톱에서는 `프로그램명 2fr / 버전 1fr / 산출물 언어 1fr` 3열 그리드를 사용한다.
- 태블릿 이하에서는 1열 스택
- 카드 상단에는 입력 목적을 설명하는 짧은 카피를 둔다

### 출력 설정 카드

포함 항목:

- `docx`, `xlsx`, `md` 형식 선택
- `클래스 다이어그램 포함` 토글

규칙:

- 포맷 선택은 칩 카드 스타일을 사용한다.
- 현재 체크박스 입력은 유지하되, 외형은 선택 카드처럼 보이게 한다.
- `클래스 다이어그램 포함`은 보조 설명이 있는 설정 행으로 보이게 한다.

### 소스 업로드 카드

포함 항목:

- 드롭존
- 업로드 힌트
- 선택된 파일 메타
- 파일 변경/제거 액션

규칙:

- 기본 상태, 드래그 오버 상태, 선택 상태, 잘못된 파일 상태를 명확히 구분한다.
- 파일이 선택되면 파일명은 줄바꿈 허용, 크기는 보조 정보로 표시한다.
- 제거 버튼은 절대 배치하되 카드 패딩 안에서 안전하게 보이도록 한다.

### 실행 액션

- `업로드` 버튼을 1차 CTA로 사용한다.
- 모바일에서는 전체 폭 버튼을 사용한다.
- 버튼 바로 아래에 입력/산출물에 대한 짧은 안내를 둔다.

## 7.2 Progress 화면

### 목표

실시간 진행 화면이 "비어 보이는 상태 페이지"가 아니라, 현재 파이프라인 단계를 신뢰감 있게 보여주는 대시보드처럼 보이게 만든다.

### 레이아웃

- 상단 `hero`:
  - 진행률 수치
  - 현재 단계명
  - 진행 바
  - 자동 이동 안내
- 하단 본문:
  - 좌측 `파이프라인 카드`
  - 우측 `상태/안내 카드`

모바일에서는 상하 스택으로 바꾼다.

### 진행률 hero

규칙:

- 퍼센트 수치는 크게 유지하되, 현재 단계와 함께 하나의 덩어리로 읽히게 한다.
- 진행 바는 과한 그라디언트 대신 안정적인 accent 색상 중심으로 표현한다.
- 단계명은 muted text보다 한 단계 강하게 보이게 한다.

### 파이프라인 카드

포함 단계:

- Intake
- Analysis
- Diagrams
- Rendering
- Packaging

규칙:

- 데스크톱에서는 현재의 가로 타임라인 패턴을 유지한다.
- 모바일에서는 세로 타임라인으로 전환한다.
- `done`, `active`, `pending`은 점, 연결선, 라벨 색상으로 모두 구분한다.

### 상태/안내 카드

규칙:

- 경고가 없을 때는 작업 상태 안내 문구를 보여준다.
- 경고가 있으면 같은 카드 안에 누적 리스트 형태로 노출한다.
- 경고가 오류처럼 과하게 보이지 않도록 경고 카드 스타일을 사용한다.

## 7.3 Result 화면

### 목표

생성 결과를 빠르게 이해하고, 묶음 다운로드와 개별 다운로드를 명확하게 수행할 수 있게 한다.

### 레이아웃

- 상단 `hero`:
  - 완료 제목
  - 짧은 설명
  - 우측 `묶음 다운로드` 1차 액션
- 요약 메타 영역:
  - `생성 시각`
  - `만료 시각`
  - `총 산출물 수`
- 조건부 경고 영역
- 산출물 카드 그리드

### 요약 메타 영역

규칙:

- `meta card` 3개를 그리드로 배치한다.
- 모바일에서는 1열 스택으로 전환한다.
- 시각 형식은 `label + strong value` 패턴을 유지한다.

### 경고 영역

규칙:

- 결과 API의 `warnings`가 비어 있지 않을 때만 렌더링한다.
- 산출물 카드와 구분되는 보조 상태 카드로 표시한다.

### 산출물 카드

구성:

- 상단: 포맷 칩, 파일 크기
- 중단: 모듈명
- 본문 강조: 파일명
- 하단: 다운로드 버튼

규칙:

- 카드형 레이아웃은 유지한다.
- hover는 약하게만 사용한다.
- 포맷 칩은 형식별 색 구분을 가지되 채도는 낮춘다.
- 파일명 영역에는 `overflow-wrap: anywhere`를 적용해 카드 폭을 넘지 않게 한다.

## 8. 데이터 및 백엔드 변경 설계

## 8.1 생성 시각 추가

현재 결과 API에는 `expiresAt`만 있고 `createdAt`이 없다. 결과 화면 상단 요약에 `생성 시각`을 넣기 위해 다음 변경이 필요하다.

- `JobRecord`에 `createdAt: Instant` 추가
- 작업 생성 시점에 `createdAt` 저장
- `JobResultResponse`에 `createdAt: Instant` 추가
- 결과 API 응답에 `createdAt` 포함

## 8.2 경고 표시 연동

현재 결과 API는 `warnings`를 반환하지만 결과 화면에서는 렌더링하지 않는다. 이를 위해 다음 작업이 필요하다.

- `result.html`에 경고 영역 추가
- `result.js`에서 `warnings` 배열을 읽어 상태 카드로 렌더링

원칙:

- 기존 `data-*` 훅은 가능한 유지한다.
- JS는 상태 토글과 렌더링 보정만 수행하고, 시각 구조는 CSS 클래스로 해결한다.
- 진행률 바 `style.width`처럼 런타임 동적 값만 인라인 스타일을 허용한다.

## 9.3 다국어 메시지

수정 대상:

- `src/main/resources/messages.properties`
- `src/main/resources/messages_en.properties`

추가 필요 문구 예시:

- 업로드 화면 보조 설명
- 진행 화면 자동 이동 안내
- 결과 화면 요약 라벨
- 경고 영역 제목
- 테마 토글 접근성 라벨

## 10. 접근성 및 반응형 검증 기준

### 접근성

- 포커스 링이 모든 주요 인터랙션 요소에서 보인다.
- 테마 토글은 현재 상태에 따라 `다크 모드로 전환` 또는 `라이트 모드로 전환` `aria-label`을 가진다.
- 드롭존은 키보드로도 파일 선택이 가능하다.
- 경고/오류 메시지는 텍스트만으로도 이해 가능하다.

### 반응형

- 375px 폭에서 수평 스크롤이 없어야 한다.
- 상단 헤더에서 브랜드와 우측 툴바가 충돌하지 않아야 한다.
- 업로드 카드, 결과 카드, 타임라인이 카드 바깥으로 넘치지 않아야 한다.

## 11. 검증 계획

### 자동 검증

- `./gradlew check`

### 수동 검증

- 브라우저에서 `http://localhost:8080/` 확인
- 업로드 → 진행 → 결과 흐름 전체 수행
- `KO / EN` 언어 전환 확인
- `light / dark` 테마 전환 확인
- 데스크톱 폭 확인
- 375px 모바일 폭 확인

### 시각 검증 체크포인트

- 헤더가 sticky로 동작하는가
- 업로드 섹션 구분이 즉시 이해되는가
- 드롭존 4개 상태가 명확히 구분되는가
- 진행 타임라인이 모바일에서 읽기 쉬운가
- 결과 요약 카드와 산출물 카드의 역할이 분명한가
- 카드나 입력 박스 사이 간격이 충분한가
- 어느 카드에서도 수평 오버플로가 발생하지 않는가

## 12. 위험 요소

- `createdAt` 추가는 프론트엔드만의 수정이 아니므로 DTO, 저장 모델, 응답 구조가 함께 변한다.
- CSS 분리 작업 중 기존 `app.css` 규칙과 충돌할 수 있다.
- 모바일 헤더 폭이 좁기 때문에 언어 토글과 테마 토글 배치가 과밀해질 수 있다.
- 진행 화면 타임라인의 데스크톱/모바일 전환이 불안정하면 상태 가독성이 떨어질 수 있다.

## 13. 구현 성공 기준

다음 조건을 모두 만족하면 이번 작업이 완료된 것으로 본다.

- 세 화면이 공통 시각 체계와 레이아웃 위계를 공유한다.
- 업로드 화면에서 입력/설정/파일 업로드/실행 흐름이 명확히 구분된다.
- 진행 화면이 비어 보이지 않고, 단계 상태를 신뢰감 있게 보여준다.
- 결과 화면 상단에 `생성 시각`, `만료 시각`, `총 산출물 수`가 노출된다.
- 결과 화면에서 경고가 있을 경우 보조 상태 카드로 표시된다.
- Bootstrap 5는 유지되며, 자동 검증과 수동 브라우저 검증을 통과한다.
