| 상태 | 초안 |

---

## 1. 문서 목적

이 문서는 `webpage-capture-tool`을 단일 캡처 실행기에서 **매뉴얼 제작 워크벤치**로 확장하기 위한 Electron 화면 초안을 정리합니다.

핵심 방향은 아래와 같습니다.

- 캡처 전 DOM 편집과 캡처 후 이미지 편집을 하나의 흐름으로 연결한다.
- Word / PPT / Markdown 산출 채널을 내보내기 단계에서 명확히 분리한다.
- 현재의 로그 중심 UI를 유지하되, 작업 대상과 결과물을 시각적으로 관리할 수 있게 만든다.

## 2. 정보 구조

### 2.1 앱 내 상위 구조

```text
App Shell
├─ 상단 바
│  ├─ 프로젝트 이름
│  ├─ 현재 모드 표시
│  ├─ 저장 / 실행 / 내보내기
│  └─ 환경 상태(브라우저, 출력 경로, 최근 실행)
├─ 좌측 탐색
│  ├─ 프로젝트
│  ├─ 캡처
│  ├─ DOM 편집
│  ├─ 이미지 편집
│  ├─ 배치 작업
│  └─ 내보내기
├─ 중앙 작업 영역
│  ├─ 웹 프리뷰
│  ├─ 이미지 캔버스
│  ├─ 테이블 리스트
│  └─ 빈 상태 화면
├─ 우측 속성 패널
│  ├─ 현재 선택된 도구 설정
│  ├─ 현재 선택된 캡처 항목 설정
│  └─ 출력 프로필 설정
└─ 하단 로그 패널
   ├─ 실행 로그
   ├─ 실패 항목
   └─ 내보내기 결과
```

### 2.2 메뉴 구조

```text
프로젝트
  - 새 프로젝트
  - 최근 프로젝트
  - 저장
  - 다른 이름으로 저장

캡처
  - 소스 입력(URL / 파일)
  - 뷰포트 프리셋
  - 캡처 범위
  - 실행 / 취소

DOM 편집
  - 텍스트 치환
  - 속성 치환
  - selector 숨기기
  - 사용자 CSS

이미지 편집
  - 크롭
  - 리사이즈
  - 블러
  - 박스 가리기
  - 텍스트 라벨
  - 강조 박스 / 화살표 / 스텝 번호

배치 작업
  - 레시피 저장
  - 레시피 불러오기
  - 선택 항목 일괄 적용
  - 실패 항목 재실행

내보내기
  - Markdown
  - Word Assets
  - PPT Assets
  - 출력 경로
  - 파일명 규칙
```

## 5. 화면 2: 캡처 설정

### 5.1 목적

- 입력 소스 설정

- 실행 시작

### 5.2 와이어프레임

```text
+--------------------------------------------------------------------------------------------------+
| 캡처                                                                                             |
+----------------------+----------------------------------------------------------------+----------+
| 프로젝트              | 소스 입력                                                       | 속성      |
| 캡처                  | -------------------------------------------------------------  |----------|
| DOM 편집              | ( ) 단일 URL                                                    | 출력 폴더  |

| 내보내기              | ( ) 목록 파일                                                   | 시트명     |

|                      |                                                                |          |
|                      | 캡처 옵션                                                       | 뷰포트     |
|                      | [Word 기본 v] [PPT 16:9 v] [Markdown 기본 v] [사용자 정의]    | 1440x1024 |
|                      |                                                                |          |
|                      | 캡처 범위                                                       | 범위       |
```

- 뷰포트 프리셋은 채널별 기본 프리셋과 사용자 정의를 함께 제공한다.

```text
| 캡처                  | -------------------------------------------------------------  |----------|
| DOM 편집              | [브라우저 프리뷰 영역]                                          | 규칙 유형   |
| 이미지 편집           |                                                                | replaceText|
| 배치 작업             |  selector hover / 클릭 시 강조                                |          |
| 내보내기              |                                                                | selector   |
|                      |                                                                | #user-name |
|                      |                                                                |          |
|                      |                                                                | 변경 값     |
|                      |                                                                | 홍길동     |
|                      |                                                                |          |
|                      |                                                                | [규칙 추가] |
+----------------------+----------------------------------------------------------------+----------+
| 규칙 목록                                                                                          |
| [on] replaceText | #user-name     | 홍길동                    | 수정 | 삭제                 |
```

### 6.3 핵심 인터랙션

- 적용 전/후 비교 토글을 제공한다.

## 13. 구현 시 고려사항

- 현재 `apps/electron/renderer/index.html`은 단일 페이지 폼 구조이므로, 좌측 탐색과 중앙 작업 영역을 가진 앱 셸 구조로 재편해야 한다.
- `renderer.js`는 기능별로 분리될 필요가 높다.
- 중앙 프리뷰는 `웹 프리뷰 모드`와 `이미지 캔버스 모드`를 전환할 수 있어야 한다.
- 메뉴를 먼저 만드는 것보다, 공통 App Shell과 상태 모델을 먼저 잡아야 이후 화면 추가 비용이 줄어든다.

## 14. 결론

Electron 화면은 기존의 “실행 폼”을 크게 유지한 채 확장하기보다, **좌측 탐색 + 중앙 워크스페이스 + 우측 속성 패널 + 하단 로그** 구조로 전환하는 것이 적합합니다.

이 구조는 다음 이점을 줍니다.

- 캡처 전 DOM 편집과 캡처 후 이미지 편집을 분리해서 이해하기 쉽다.
- 레시피와 배치 작업을 독립된 개념으로 다룰 수 있다.
- Word / PPT / Markdown 출력 정책을 마지막 단계에서 명확히 노출할 수 있다.
