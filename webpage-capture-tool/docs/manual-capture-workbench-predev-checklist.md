## 1. 문서 목적

이 문서는 `Manual Capture Workbench` MVP를 구현하기 전에 합의해야 하는 결정 항목을, 실제 구현 순서에 맞춰 다시 분류한 체크리스트입니다.

이 문서의 목적은 아래와 같습니다.

- 구현 도중 정책이 바뀌어 재작업이 생기는 지점을 줄인다.
- `packages/core`, `packages/cli`, `apps/electron` 사이의 계약을 먼저 고정한다.
- 모든 항목을 한 번에 잠그지 않고, 실제로 필요한 시점에만 결정을 강제한다.
- 특히 재작업 비용이 큰 `좌표계`, `IPC 계약`, `파일 생명주기`, `내보내기 계약`을 초기에 잠근다.

---

## 2. Gate 운영 원칙

이 체크리스트는 두 단계 게이트로 운영합니다.

- `Gate A. 착수 전 확정`
  - Stream A, B에 본격 착수하기 전에 잠가야 하는 공통 계약이다.
  - 이 단계가 흔들리면 이후 여러 Stream이 동시에 다시 수정된다.
- `Gate B. Stream 전 확정`
  - 해당 Stream에 들어가기 직전에만 잠가도 되는 결정이다.
  - 다른 Stream을 막지 않도록 범위를 분리한다.

운영 규칙:

- Gate A 항목은 체크 완료 전까지 구현 착수 금지 대상으로 본다.
- Gate B 항목은 해당 Stream 시작 전까지 체크 완료하면 된다.
- 구현 중 새 요구가 생겨도 먼저 이 문서를 갱신한 뒤 진행한다.
- 아래 조건 중 하나에 해당하면 Gate B 항목이라도 Gate A로 승격한다.
  - 저장 포맷을 바꾼다.
  - 좌표 저장 기준을 바꾼다.
  - CLI / Core / Electron 간 데이터 계약을 바꾼다.
  - 이미 생성된 산출물의 재현성을 깨뜨린다.

## 4. Gate A. 착수 전 확정

### A-01) 프로젝트 JSON 최종 스키마

- [ ] `projectVersion` 필드 포함 여부를 결정했다.
- [ ] `sources`, `capturePreset`, `domRules`, `editRules`, `exportProfiles`, `runHistory`의 필수/선택 여부를 결정했다.
- [ ] 페이지 단위 override 구조를 MVP에 포함할지 결정했다.
- [ ] 하위 호환이 깨질 때의 마이그레이션 전략을 결정했다.

추천 방향:

- `projectVersion`은 semver 문자열(`"1.0.0"`)로 포함한다.
- `name`만 필수로 두고, 나머지는 기본값을 가진 선택 필드로 처리한다.
- 페이지 단위 override는 MVP에서 제외한다. PRD상 P1이며 스키마와 UI 복잡도를 크게 높인다.
- 마이그레이션 함수는 `packages/core/src/recipe-store.js`에 둔다.
- 저장 포맷은 “열기 시 마이그레이션, 저장 시 최신 버전으로 승격” 원칙을 사용한다.

### A-02) 입력 소스 파일 파싱 규칙

- [ ] 단일 URL 입력과 파일 입력이 같은 내부 데이터 모델로 수렴하는지 결정했다.
- [ ] xlsx/csv에서 URL 컬럼 자동 인식 규칙을 결정했다.
- [ ] 빈 행, 주석 행(`#` 시작), 중복 URL 처리 방식을 결정했다.
- [ ] 파싱 실패 시 사용자 안내 형식을 결정했다.
- [ ] xlsx 파싱 라이브러리를 확정했다.

추천 방향:

- 내부 모델은 `sources[]`로 통일하고, 단일 URL도 길이 1인 source 배열로 변환한다.
- URL 컬럼 자동 인식은 `url`, `URL`, `link`, `주소`, `detailPage` 우선 탐색 후, 없으면 사용자가 UI에서 지정하도록 한다.
- TXT는 현재 동작처럼 `#` 주석과 빈 줄을 무시한다.
- 중복 URL은 즉시 제거하지 말고, “원본 rows 유지 + 실행 대상 dedupe 결과 별도 보관”으로 분리한다.
- MVP에서는 새 라이브러리로 바꾸지 않고 기존 `xlsx` 기반 파서를 유지한다. 실제 대용량 병목이 확인될 때만 `exceljs`로 전환을 검토한다.
- 파싱 실패 시 UI에는 친화적 메시지와 샘플 형식을 함께 보여주고, 디버그 로그에는 발견 헤더와 파일 경로를 남긴다.

결정 메모:

### A-03) 뷰포트 프리셋 규격 정의

- [ ] Word, PPT, Markdown 기본 프리셋 수치를 확정했다.
- [ ] 프리셋과 사용자 정의 입력의 공존 방식을 결정했다.
- [ ] `fullPage` 캡처에서 높이 값이 어떻게 동작하는지 문서화했다.

추천 방향:

- Word 기본: `1440 x 1024`
- PPT 16:9: `1920 x 1080`
- Markdown 기본: `1280 x 800`
- 프리셋 선택 후 width/height 직접 수정 가능하게 한다.
- `fullPage`에서는 height가 “초기 viewport 높이”로만 동작하고, 최종 이미지 높이는 페이지 전체 길이에 따라 결정됨을 UI에 명시한다.
- PRD의 P0 범위에서는 `16:9`, `4:3`, `자유 비율`, `원본 유지`를 캡처 옵션이 아니라 크롭 보조 프리셋으로 취급할지 별도 메모를 남긴다.

결정 메모:

### A-04) 캡처 범위와 좌표 기준

- [ ] `fullPage`, `viewport`, `selector` 캡처의 입력 옵션과 출력 결과를 문서로 고정했다.
- [ ] selector 캡처의 padding 확장 허용 여부를 결정했다.
- [ ] 저장용 좌표의 기준 좌표계를 하나로 고정했다.
- [ ] DOM 단계 좌표와 이미지 편집 단계 좌표의 관계를 정의했다.

추천 방향:

- 캡처 범위는 `fullPage`, `viewport`, `selector` 세 가지로 고정한다.
- selector 캡처는 `paddingPx` 옵션을 지원하되 기본값은 `0`으로 둔다.
- 저장용 정규 좌표계는 `rawImagePx`로 통일한다.
- 좌표계는 아래 네 종류를 구분하고, 영구 저장은 `rawImagePx`만 허용한다.

| 좌표계 | 의미 | 저장 여부 |
|---|---|---|
| `pageCssPx` | Playwright 페이지의 DOM/CSS pixel 좌표 | 저장하지 않음 |
| `rawImagePx` | 캡처 직후 원본 PNG 기준 pixel 좌표 | 저장함 |
| `displayPx` | Electron 캔버스 표시 좌표 | 저장하지 않음 |
| `exportImagePx` | crop/resize 후 최종 산출물 좌표 | 저장하지 않음 |

- DOM selector bbox는 실행 시점에만 `pageCssPx -> rawImagePx`로 변환한다.
- crop, blur, box 등 이미지 편집 규칙은 모두 `rawImagePx` 기준으로 저장한다.
- device scale factor는 기본값을 명시적으로 지정하고, OS 기본값에 묶이지 않게 한다.

결정 메모:

### A-05) Electron 렌더러 모듈화 전략

- [ ] 렌더러 분리 단위를 화면 중심으로 할지 기능 중심으로 할지 결정했다.
- [ ] 번들러 도입 여부를 결정했다.
- [ ] 현재 프로젝트의 CommonJS 중심 스타일과 충돌하지 않는 방식을 선택했다.

추천 방향:

- MVP에서는 번들러를 도입하지 않는다.
- `main.js`, `preload.js`, `packages/*`는 기존 CommonJS 스타일을 유지한다.
- renderer는 브라우저 안전 코드만 두고, `renderer.js`를 진입점으로 하되 화면/기능별 파일로 분리한다.
- 분리 단위는 `capture-screen`, `dom-screen`, `image-screen`, `export-screen`, `shared-state`처럼 화면 기준을 우선한다.
- 렌더러에서 Node API 직접 접근을 늘리지 말고, 상태 렌더링과 이벤트 바인딩을 분리한다.

결정 메모:

### A-06) Electron IPC 채널 명세와 preload 브리지 계약

- [ ] IPC 채널 네이밍 컨벤션을 결정했다.
- [ ] preload 브리지에서 노출할 API 목록을 초안 수준으로 확정했다.
- [ ] 요청/응답과 진행 상태 스트리밍의 기준을 결정했다.
- [ ] renderer가 채널 문자열을 직접 알지 않아도 되게 구조를 정했다.

추천 방향:

- 채널 네이밍은 `{기능}:{동작}` 형식으로 통일한다.
- 예시: `capture:run`, `capture:cancel`, `project:save`, `project:open`, `export:run`
- 요청-응답은 `ipcRenderer.invoke` / `ipcMain.handle`를 사용한다.
- 진행 상태 스트리밍은 `webContents.send` / `ipcRenderer.on`을 사용한다.
- preload는 `window.captureApi` 또는 `window.workbenchApi` 하나의 네임스페이스로 묶고, 채널 문자열은 preload 내부에만 둔다.
- 신규 기능이 늘어나도 Electron main이 CLI stdout 문자열 파싱에 과도하게 의존하지 않도록, 가능하면 `packages/core`의 구조화된 결과 객체를 직접 주고받는 방향으로 설계한다.

결정 메모:

### A-07) 배치 실행 취소와 실행 소유권 정책

- [ ] 실행 취소 시 현재 처리 중인 항목을 완료하고 중단할지, 즉시 중단할지 결정했다.
- [ ] 취소 후 재실행 범위를 결정했다.
- [ ] 실행 흐름의 소유권을 `CLI child process 래핑`에 둘지 `core job runner`에 둘지 결정했다.
- [ ] 향후 진행률, 재시도, 후처리 연결을 어느 계층에서 담당할지 결정했다.

추천 방향:

- 취소는 “현재 항목 완료 후 중단”을 기본 정책으로 한다.
- 취소 후 재실행은 미완료 항목만 대상으로 한다.
- 장기적으로는 `packages/core`에 협조적 취소가 가능한 job runner를 두고, CLI와 Electron이 같은 실행기를 호출하는 구조를 목표로 한다.
- MVP 초기에 Electron이 CLI를 감싸는 구조를 유지하더라도, 신규 기능의 결과 모델은 stdout 텍스트가 아니라 구조화된 이벤트/객체 중심으로 옮겨 간다.
- 현재 `currentChild.kill()` 같은 즉시 종료 방식은 임시 대응으로만 사용하고, 최종 정책으로 고정하지 않는다.

결정 메모:

### A-08) 중간 캡처 파일과 최종 내보내기 파일 관리 정책

- [ ] 캡처 원본 파일, 편집 후 파일, 최종 내보내기 파일의 저장 위치를 결정했다.
- [ ] 세션 종료 / 프로젝트 닫기 / 프로젝트 삭제 시 정리 정책을 결정했다.
- [ ] 프로젝트 JSON에 무엇을 저장하고 무엇을 저장하지 않을지 결정했다.

추천 방향:

- 캡처 원본과 편집 결과는 사용자 출력 폴더가 아니라 앱 전용 cache 디렉터리에 저장한다.
- 기본 경로는 `app.getPath("userData")/manual-capture-cache/{projectId}/{sessionId}/` 같은 persistent cache를 사용한다.
- 프로젝트 JSON에는 임시 절대 경로를 직접 저장하지 않고, `assetId`, `cacheKey`, `sourceHash` 같은 식별자만 저장한다.
- 최종 내보내기 파일만 사용자 지정 출력 경로에 쓴다.
- “캐시 재사용 가능”과 “캐시가 없어도 원본 source + 규칙으로 재생성 가능”을 함께 만족시키는 구조를 목표로 한다.

#### B-C02) DOM 규칙 적용 타이밍과 재시도 정책

- [ ] 페이지 로드 완료 기준을 결정했다.
- [ ] SPA selector 재시도 횟수와 간격을 결정했다.
- [ ] DOM 규칙 실패가 전체 실패인지 부분 실패인지 결정했다.
- [ ] headless/headed 차이 허용 범위를 결정했다.

추천 방향:

- 기본 waitUntil은 `networkidle`로 둔다.
- 사용자가 `load`, `domcontentloaded`, 추가 대기(ms)를 조정할 수 있게 한다.
- 재시도는 기본 3회, 500ms 간격으로 둔다.
- DOM 규칙 실패는 기본적으로 부분 실패이며, 경고 로그를 남기고 캡처는 계속 진행한다.
- headless/headed 결과 차이는 “허용”이 아니라 “문서화해야 하는 예외”로 본다.

결정 메모:

## 7. 권장 착수 순서

1. Gate A를 먼저 잠근다.
2. Stream A(App Shell)와 Stream B(캡처 옵션)를 시작한다.
3. Stream C 진입 직전에 DOM 규칙 3개 항목을 잠근다.
4. Stream D 진입 직전에 이미지 편집 3개 항목을 잠근다.
5. Stream E, F, G는 각 단계 직전에 레시피 / export / 로그 계약을 잠근다.

이 순서를 따르면 “문서 결정 때문에 모든 작업이 멈추는 상황”을 줄이면서도, 재작업 비용이 큰 계약은 초기에 보호할 수 있다.
